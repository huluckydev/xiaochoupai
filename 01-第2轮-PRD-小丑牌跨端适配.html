三、核心设计原则

P1 · 强制横屏，不做竖屏适配

游戏仅支持横屏布局。五端全部通过各自的平台机制锁定横屏方向。竖屏状态下，游戏内容不渲染，显示竖屏挡板（已在第 1 轮完成）。

决策依据：Balatro 原作是横屏游戏，现有 UI 布局（左 sidebar + 右三段式主区）为横屏设计，强行适配竖屏需要重新布局，属于游戏 UI 重构，超出本轮范围。

P2 · 只改 UI，不动游戏逻辑

以下内容在本轮任何情况下都不允许修改：

  • 牌型识别逻辑（getHandRank 等函数）
  • 计分公式（chips / mult 计算链）
  • 游戏状态机（playing / shop / won / lost）
  • Joker 效果定义与触发顺序
  • AI 出牌启发式枚举逻辑
  • 商店随机化逻辑

P3 · 网页版兼容保底

所有新增 UI 适配代码必须满足：

  • Tauri 专属 API（如 Tauri JS API、原生状态栏控制）必须用 window.__TAURI__ 存在性判断包裹，Web 环境不触发
  • CSS 安全区变量（env(safe-area-inset-*)）在不支持的浏览器下优雅降级为 0
  • GitHub Pages 部署的 Web 版体验不因本轮改动劣化

P4 · 适配粒度：UI 感知，不做原生控件重写

本轮适配停留在"CSS + Tauri 配置层"，不涉及引入原生 UI 组件库、重写渲染引擎或大幅重构 HTML 结构。目标是让现有的 Vue 3 单文件在各端"看起来对、摸起来顺"，而不是"重做一次 App"。

四、功能需求

F1 · 强制横屏锁定 P0

各平台实现方式（需求层描述，技术方案由工程师决定）：

平台需求描述期望结果
Android 在 Tauri 的 Android 配置（AndroidManifest.xml）中声明 Activity 的 screenOrientation 为 landscape 或 sensorLandscape 用户无论如何旋转设备，App 始终保持横向显示，系统不允许自动旋转
iOS 在 Tauri 的 iOS 配置（Info.plist）中设置 UISupportedInterfaceOrientations 仅包含 Landscape Left / Landscape Right App 始终横屏，竖握手机时系统层面不旋转 UI
macOS 桌面端无方向锁定概念，窗口横向布局即可。在 Tauri 配置中设置窗口初始宽度 > 高度（横向比例） 启动时窗口为横向，用户手动拖拽为竖向时，游戏区仍按横屏布局渲染（不做竖屏特殊处理）
Windows 与 macOS 一致，Tauri 配置窗口初始横向比例 同上
Web 已通过 CSS @media (orientation: portrait) + 竖屏挡板实现（第 1 轮已完成） 竖屏时显示挡板，横屏时正常渲染

F2 · 竖屏挡板 已完成，确认保留 P0

第 1 轮已完成以下内容，本轮确认保留，无需修改：

  • CSS @media (orientation: portrait) 检测竖屏
  • 竖屏时覆盖层显示提示文字（"请旋转屏幕以横屏游玩"或类似措辞）
  • 游戏区内容在竖屏时不渲染或不可交互
确认事项 在 Tauri 封装的 App 内，WebView 同样支持 CSS media query，竖屏挡板逻辑无需改动。但需验证 Android/iOS 在 Tauri WebView 内 orientation 媒体查询的行为与普通浏览器一致。若存在差异，由工程师阶段处理。

F3 · 安全区适配（刘海屏、Home Indicator） P0

3.1 适用平台

Android 刘海屏（notch）、打孔屏、底部手势条
iOS Dynamic Island / 刘海（iPhone X 及以后）、Home Indicator（底部横条）

3.2 需求描述

游戏 UI 的所有可交互元素（按钮、手牌、Joker 卡、sidebar 面板）必须落在安全区内，不被刘海/打孔/Home Indicator 遮挡。

区域风险元素（可能被遮挡）适配要求
左侧（Android notch 可能出现在横屏左侧） 左 sidebar 顶部 logo、盲注面板 左 padding 增加 env(safe-area-inset-left)，确保内容不进入刘海区域
右侧（部分机型摄像头在右侧） 右主区手牌操作按钮、牌堆 右 padding 增加 env(safe-area-inset-right)
底部（iOS Home Indicator / Android 手势条） 底部出牌/弃牌按钮行 底部 padding 增加 env(safe-area-inset-bottom)，按钮不被手势条覆盖
顶部（Android 状态栏） 右上角设置按钮 ⚙️、Joker 区标题 顶部 padding 增加 env(safe-area-inset-top)，或通过全屏模式消除状态栏（见 F6）

3.3 实现方式说明（需求层）

CSS 层使用 padding: env(safe-area-inset-top) env(safe-area-inset-right) env(safe-area-inset-bottom) env(safe-area-inset-left) 应用于顶层容器，或针对各区域单独注入。具体哪个 DOM 元素挂这个 padding 由工程师决定。

对于不支持 env() 的环境（旧浏览器、桌面端），安全区变量自动降级为 0，不影响显示。

HTML <meta name="viewport"> 需增加 viewport-fit=cover，否则 iOS Safari / WKWebView 不会暴露安全区 CSS 变量。

注意 安全区 padding 不能简单加在 body 上了事。现有布局是 100vh 全屏 + overflow: hidden，需要在不破坏原有 100vh 充满效果的前提下注入安全区。工程师阶段需要验证加了 safe-area padding 后整体布局不错位、不产生滚动条。

4.2 消除移动端点击延迟

移动端浏览器（及 Tauri WebView）默认有 300ms 点击延迟。需在 CSS 中添加 touch-action: manipulation 到所有可交互元素，或全局应用到 body，消除点击延迟，使响应与桌面端一致。

4.3 禁止不必要的触摸行为

行为要求原因
页面双指缩放禁止（user-scalable=no）游戏界面不是网页内容，缩放会破坏布局
页面级滚动（上下滑）禁止（overflow: hidden 已设）已在第 1 轮处理，确认保留
长按弹出系统菜单（文字选中、图片保存）禁止长按扑克牌不应弹出"复制"、"查看图片"等系统菜单，干扰游戏操作
橡皮筋弹性滚动（iOS overscroll）禁止避免游戏区域被拉扯产生空白

4.4 手势说明

本轮游戏仅需"点击"操作（选牌、点按钮）。不引入滑动手势、拖拽手势、捏合手势——这属于玩法层改动，超出本轮范围。

5.2 字体渲染要求

现有字体使用规则（Press Start 2P / VT323 / Inter + PingFang SC）在第 1 轮已锁定，保持不变。

DPI 适配要求：

  • 字体单位统一使用 px（当前实现），CSS 层由浏览器/WebView 自动处理 DPI 缩放，无需手动换算
  • Tauri WebView 在各平台启用 viewport 元标签的 DPI 感知（width=device-width, initial-scale=1，已在 index.html 中配置），确认保留
  • Google Fonts 加载（Press Start 2P / VT323 / Inter）：App 版需确认在无网络时的降级字体（system-ui 作为 fallback）不破坏布局

5.3 元素缩放策略

游戏 UI 使用固定 px 尺寸 + 动态缩放 CSS（第 1 轮已实现 --scale 动态缩放变量）。桌面端窗口缩小时，整体 scale() 等比缩放，不做响应式重排。这个策略在各端均适用，无需为 DPI 单独设计另一套缩放逻辑。

关于离线字体 若后续要求 App 完全离线可用，需将三种字体文件打包进 Tauri App 资源。本轮仅提出需求，离线字体打包由工程师在 Tauri 框架配置阶段处理。

F6 · 平台原生状态栏处理 P0（移动端）

适用平台：Android iOS

6.1 需求目标

游戏进入后，系统状态栏（显示时间、电量、信号等）应隐藏或沉浸式叠加，游戏 UI 占据全部屏幕面积，不留黑条/灰条。

6.2 各平台要求

平台要求说明
Android 启用 Immersive Mode（沉浸式全屏），隐藏状态栏和导航栏 通过 Tauri Android 插件或 AndroidManifest 配置。用户从屏幕边缘下滑可临时呼出系统 UI，离开焦点后自动收回（Sticky Immersive）
iOS 隐藏状态栏（Status Bar Hidden），或设为 Overlay 叠加模式（状态栏透明叠在游戏内容上） 通过 Info.plist 的 UIStatusBarHidden 或 Tauri iOS 插件配置。全屏游戏通常选择完全隐藏

6.3 与安全区的关系

若启用全屏隐藏状态栏，env(safe-area-inset-top) 值会变为 0（状态栏不再占空间），但刘海/Dynamic Island 区域的 safe-area-inset-left / safe-area-inset-right 在横屏下仍有值。工程师需注意两者联动。

F7 · 窗口尺寸适配（桌面端） P1

适用平台：macOS Windows

7.1 最小窗口尺寸

参数要求值依据
最小宽度1024px现有 UI 8 张手牌并排（100px × 8 + gap 8px × 7 = 856px）+ 左 sidebar（280px 最小）共约 1136px；考虑整体缩放，以 1024px 为最小可用边界

7.2 窗口缩放响应

桌面端用户可能调整窗口大小。游戏 UI 应：

  • 窗口大于推荐尺寸时：游戏区居中显示，或等比放大填满（由工程师决定哪种方案，PRD 不强制）
  • 窗口小于推荐尺寸但大于最小尺寸时：整体等比缩放（scale()），不裁切、不重排
  • 窗口小于最小尺寸时：Tauri 配置层阻止进一步缩小（不允许用户拖到最小尺寸以下）
  • 窗口宽度 < 高度时（极端竖向拖拽）：显示竖屏挡板（与移动端逻辑一致）

7.3 全屏模式（可选，P2）

桌面端可通过 Tauri 提供"全屏游玩"按钮（菜单或快捷键），进入真全屏模式。这是 nice-to-have，本轮不做，记录到 P2 待办。

八、验收标准

说明 以下验收清单在 Tauri 框架搭建完成、各平台 App 可运行后执行。本轮（UI 需求定义阶段）不可执行，记录在此供后续轮次使用。

8.1 Android 验收

  • 安装 APK 后，App 图标正常显示
  • 启动后自动横屏显示，系统不允许旋转为竖屏
  • 竖握手机时（设备物理竖向），App 保持横屏，不显示竖屏挡板（因为系统已锁定横屏，CSS media query 不触发 portrait）
  • 刘海屏 / 打孔屏设备：左侧 sidebar 内容不被摄像头遮挡
  • 底部手势条区域：出牌/弃牌按钮不被手势条覆盖，按钮完整可点
  • 系统状态栏隐藏（或沉浸叠加），游戏占满全屏
  • 点击手牌、按钮无明显 300ms 延迟
  • 长按手牌不弹出系统菜单（无"复制"、"粘贴"等）
  • 上下滑动无页面弹性滚动效果
  • 双指捏合无缩放效果
  • 冷启动到可交互 ≤ 3 秒
  • APK 大小 ≤ 30MB
  • 出牌动画流畅（主观可接受，以 30fps 为底线）

8.2 iOS 验收

  • App 启动后横屏显示，竖握手机时 App 保持横屏
  • iPhone 有刘海 / Dynamic Island：横屏时左侧 sidebar 内容不进入刘海遮挡区
  • 底部 Home Indicator：出牌按钮不被覆盖
  • 状态栏隐藏，游戏区占满屏幕
  • 点击无延迟，长按无系统菜单
  • 页面不可上下滑动（overscroll 禁止）
  • 双指捏合无缩放
  • 冷启动 ≤ 3 秒
  • IPA 大小 ≤ 30MB

8.3 macOS 验收

  • App 启动时窗口为横向（宽 > 高），默认尺寸约 1440×900
  • 窗口不可缩小到 1024×600 以下（Tauri 阻止）
  • 将窗口缩小到 1200×700：游戏 UI 整体等比缩小，不产生滚动条，不裁切内容
  • 将窗口拖为极端竖向（宽 < 高）：显示竖屏挡板提示
  • hover 样式正常（按钮 hover 上移 + 高亮，Joker 卡 hover 有反馈）
  • Retina 显示器上字体清晰，无模糊
  • 非 Retina 显示器（1× 外接显示器）上 UI 正常显示
  • 冷启动 ≤ 2 秒
  • .app 大小 ≤ 20MB

8.4 Windows 验收

  • App 启动时窗口为横向，默认约 1440×900
  • 窗口不可缩小到 1024×600 以下
  • 缩小窗口：UI 等比缩放，不产生滚动条
  • 极端竖向拖拽：显示竖屏挡板
  • 系统 DPI 设为 125% / 150%：UI 不模糊，文字清晰
  • hover 样式正常
  • 冷启动 ≤ 2 秒
