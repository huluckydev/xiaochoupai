1

设计气质定调

移动端横屏 · 三个形容词

克制 · 沉浸 · 精准

  • 克制：移动横屏空间有限（高度约 360–430px 逻辑像素），不增加新 UI 层级，不新增浮层遮挡，以"缩放而非重排"应对空间压缩。保留 PC 版所有信息密度，不做删减。
  • 沉浸：延续第 1 轮的深蓝水纹 Balatro 风格，全屏无系统干扰（沉浸式全屏 / 隐藏状态栏），刘海、Home Indicator 安全区均用留白而非遮挡来处理，不破坏视觉完整性。
  • 精准：所有可交互元素（手牌、按钮、Joker 卡）触控热区在缩放变换后仍满足 44pt/dp 最小规格，不因 CSS scale 导致实际触控区域缩水。触控反馈即时（消除 300ms 延迟），长按无系统菜单干扰。

选择"缩放适配"而非"响应式重排"的原因：游戏 UI 是固定像素设计（元素位置关系强耦合），重排会破坏出牌动画中的空间映射逻辑（GSAP 动画依赖元素绝对坐标飞字）。等比缩放可以零改动地保留所有动画，且第 1 轮已实现 --scale 变量机制。

3

整体布局方案

scale 缩放适配 · 不重排

3.1 布局策略

保持 PC 版 HTML 结构（sidebar + main-area 三行网格）不变，在移动端横屏时对根容器应用 transform: scale(0.51) 并调整 transform-origin，使整体等比缩放至目标视口。

为什么不用响应式重排
  • GSAP 动画（飞牌、飞字、Joker 触发）依赖元素的 getBoundingClientRect() 坐标，重排会导致所有动画坐标计算失效，修复成本极高
  • 第 1 轮已实现 --scale 动态缩放机制，本方案沿用扩展
  • scale 缩放保留所有 hover/active 样式、过渡动画、布局比例，零额外开发

4.2 CSS 变量方案

安全区方向CSS 变量应用位置风险元素降级值
左侧（刘海 / 打孔） env(safe-area-inset-left) 根容器 .game-wrap 的 padding-left 或 margin-left 左 sidebar 顶部 logo、盲注面板顶部 0（旧浏览器正常显示）
右侧（右侧打孔屏） env(safe-area-inset-right) 根容器 .game-wrap 的 padding-right 右侧操作按钮边缘、设置按钮 0
底部（Home Indicator / 手势条） env(safe-area-inset-bottom) .hand-zone 的 padding-bottom 出牌/弃牌按钮行（.action-row） 0
顶部（状态栏，全屏模式后为 0） env(safe-area-inset-top) 全屏模式隐藏状态栏后此值 = 0，无需额外处理 右上角 ⚙️ 设置按钮 0
注意：安全区与 scale 的叠加关系

安全区 padding 应加在 scale 容器的父级（#app 或 .safe-wrapper），而非 scale 容器本身。若加在 scale 容器内部，padding 值也会被 scale 压缩（例如 47px × 0.51 = 24px，小于实际刘海宽度，无法有效保护）。

正确做法：在 #app 层应用 padding-left: env(safe-area-inset-left)，让 .game-wrap（缩放容器）从安全区内边界开始布局。

4.3 viewport meta 配置

现有 index.html 已包含 viewport-fit=cover，确认保留，无需修改：

<meta name="viewport" content="width=device-width, initial-scale=1.0, viewport-fit=cover, maximum-scale=1.0, user-scalable=no">

viewport-fit=cover 是 iOS Safari / WKWebView 暴露安全区 CSS 变量的必要条件。Android Tauri WebView 同样支持此属性。

5

按钮区设计规范

触控区域 · 安全区保护 · 不遮挡游戏内容

5.1 按钮布局原则

核心原则：按钮有独立区域，不遮挡游戏内容

操作按钮（出牌/弃牌/排序/AI出牌）位于 .action-row，固定在手牌区（.hand-zone 第三行）底部，属于手牌区内部布局，不使用 position: fixed 悬浮覆盖，不遮挡小丑牌区、出牌区和手牌显示区。

5.2 缩放后按钮触控热区计算

现有 .px-btn 规范：min-height: 52px。经过 scale 0.51 缩放后，视觉高度约 26.5px，低于 44pt 标准。

scale 缩放导致触控区域收缩问题

transform: scale() 会同时缩小视觉大小和触控热区（touch target）。若对整个游戏容器应用 scale(0.51)，按钮的实际触控面积 = 原始 CSS 面积 × 0.51²，远小于 44pt 要求。

两种解决方案对比：

方案描述优点缺点推荐
方案 A：伪元素扩展热区 对 .action-row .px-btn 添加 ::before 伪元素，扩大至真实 44pt/dp（约 86px 原始 CSS 尺寸，缩放后约 44px） 不改变视觉，触控准确 伪元素尺寸需精确计算：44 ÷ 0.51 ≈ 86px 推荐
方案 B：action-row 不参与 scale 将 .action-row 从 scale 容器中提取，使用 position: fixed 固定在底部，不参与整体缩放 触控区域 = 原始 CSS 尺寸，无需换算 按钮位置从 grid 流中脱离，可能遮挡内容；需精确计算 fixed 坐标 不推荐（违反"不遮挡游戏内容"原则）
方案 C：仅对按钮行 scale-up 在 .action-row 上应用反向 scale（scale(1/0.51)），使其在整体 scale 容器内视觉恢复原尺寸 触控区域自动正确，无需计算 按钮视觉变大，会侵占手牌区空间 备选

字体与 DPI 适配说明

像素字体 · 多倍屏渲染 · 降级预期

8.1 字体三类分工（继承第 1 轮规范）

字体用途移动端表现预期降级字体
Press Start 2P 仅纯英文装饰：Logo、JOKERS 段标题、chips/mult 数字块 高 DPI（2×/3×）下字体锐利，像素点阵感强。缩放后字号变小（如原 14px × 0.51 ≈ 7px），在 3× 屏仍清晰可辨，在 1.5× 屏可能偏糊 monospace（无像素风格，但不破坏布局）
VT323 所有数字大屏：Round Score、金币、Hands/Discards、目标分 液晶字感在高 DPI 下更纯粹。缩放后数字仍可辨认（VT323 设计为大字号使用，缩放至约 20px 仍可读） monospace
Inter + PingFang SC 所有中文标签、按钮、说明文字 系统字体渲染最优，各 DPI 下均清晰。缩放后中文字号约 7px（原 14px × 0.51），接近极限可读边界，但在 3× Retina 上实际物理像素约 21px，仍清晰 system-ui, sans-serif（中文 fallback 自动使用系统中文字体）

8.2 各平台 DPI 下的字体渲染预期

平台DPR缩放后 7px 中文的实际物理像素可读性处理建议
iPhone Retina 3× 7 × 3 = 21 物理像素 清晰可读 无需额外处理
iPhone SE（2×） 2× 7 × 2 = 14 物理像素 勉强可读（偏小） 可适当提高 scale 值至 0.55（667px 设备更宽裕）
Android xhdpi 2× 7 × 2 = 14 物理像素 勉强可读 同上，scale 可动态调整
Android xxhdpi 3× 7 × 3 = 21 物理像素 清晰可读 无需额外处理
Android xxxhdpi（旗舰） 4× 7 × 4 = 28 物理像素 非常清晰 效果最佳
动态 scale 值建议

工程师实现时，建议根据 viewport 宽度动态计算 scale 值（而非固定 0.51），以适应不同宽度的 Android 设备：

  • 基准设计宽度：1440px
  • scale = (viewportWidth - safeLeft - safeRight) / 1440
  • 最小值 clamp：0.45（避免过小）
  • 最大值 clamp：0.70（避免桌面端触发移动端样式后过大）

8.3 字体平滑渲染

在移动端 WebView 中添加以下 CSS 改善字体渲染质量：

@media (max-width: 1024px) and (orientation: landscape) {
  * {
    -webkit-font-smoothing: antialiased;
    -moz-osx-font-smoothing: grayscale;
  }
}

8.4 离线字体降级预期

若 App 在无网络环境下无法加载 Google Fonts，降级到 monospace / system-ui。视觉影响评估：

  • Press Start 2P 降级：像素字体变为普通等宽字体，像素游戏风格弱化，但 Logo 和段标题布局不破坏
  • VT323 降级：数字显示变为普通等宽字体，液晶数字感消失，但可读性不变
  • Inter 降级：中文 fallback 到 PingFang SC / Microsoft YaHei（已在 font-family 中声明），几乎无视觉差异

PRD 倾向于在 Tauri 打包时内嵌字体文件，本设计规范不强制要求，由工程师在 Tauri 配置阶段决定。

9.1 断点定义

断点条件触发场景处理策略
移动端横屏 @media (max-width: 1024px) and (orientation: landscape) iPhone / Android 横屏（780–960px 宽） 启用 scale 缩放 + 安全区 padding + 触控优化

隔离原则

所有移动端横屏适配样式必须写在 @media (max-width: 1024px) and (orientation: landscape) 块内。PC 版（宽 > 1024px）绝对不触发这些样式，确保 PC 体验零变化。

9.2 scale 计算示意（JavaScript 逻辑描述）

/* 移动端横屏 scale 动态计算（由工程师决定具体实现方式）*/
const BASE_WIDTH = 1440; // 设计基准宽度
const safeLeft = parseInt(getComputedStyle(document.documentElement)
                   .getPropertyValue('--safe-left')) || 0;
const availableWidth = window.innerWidth - safeLeft - safeRight;
const scale = Math.min(0.70, Math.max(0.45, availableWidth / BASE_WIDTH));
document.querySelector('#app').style.setProperty('--scale', scale);

设计决策说明

关键选择的理由

决策 A：等比 scale 缩放，而非响应式重排

结论

选择 scale 缩放（transform: scale）而非响应式 CSS 重排（媒体查询重新布局）。

  • 理由 1：GSAP 出牌动画（飞牌、飞字、Joker 触发粒子）依赖 getBoundingClientRect() 的绝对坐标映射，响应式重排会导致所有动画坐标计算失效，修复成本等于重写整个动画系统
  • 理由 2：游戏布局各元素位置关系强耦合（牌堆 absolute 定位、出牌区尺寸与手牌选中动画幅度耦合），重排破坏这些隐式约束
  • 理由 3：第 1 轮已实现 --scale CSS 变量 + scale() 机制，本方案扩展现有实现，工作量最小
  • 代价：按钮触控热区随 scale 缩小，需要额外的 ::before 伪元素扩展（见 §5）。这是可接受的工程代价

决策 B：安全区加在 scale 父级（#app），而非 scale 容器内部

结论

安全区 padding 加在 #app（scale 容器的父级），而非 .game-wrap（scale 容器本身）。

  • 理由：若安全区 padding 加在 scale 容器内部，这些 padding 值也会被 scale 压缩。例如 47px 左安全区 × 0.51 ≈ 24px，小于实际刘海宽度（约 47px），内容仍会被遮挡
  • 正确做法：#app 层（不参与 scale）应用安全区，将可用空间的物理边界确定后，再让 scale 容器在这个安全空间内布局

决策 C：设置按钮（⚙️）脱离 scale 容器

结论

建议将右上角 ⚙️ 设置按钮设为 position: fixed，脱离 scale 容器。

  • 理由 1：脱离 scale 后，设置按钮的触控区域保持原始 44×44px CSS 尺寸，无需 ::before 扩展
  • 理由 2：设置 Modal 本身应覆盖全屏（fixed 定位），设置按钮也是 fixed 更自然
  • 理由 3：第 1 轮规范已定义设置按钮为 position: absolute; top: 16px; right: 16px; z-index: 100，改为 fixed 且加 safe-area-inset-right 是最小改动
  • 注意：fixed 定位在 scale 容器的 transform 上下文中会相对于 viewport，这反而符合需求

决策 D：竖屏挡板不做简化，保留动画

结论

竖屏挡板保留 📱 摆动动画，不因移动端性能顾虑而简化为静态。

  • 理由：竖屏挡板只是一个纯 CSS keyframe 动画（rotate），CPU 占用极低。竖屏时游戏区已隐藏（display: none），GSAP 动画暂停，整体性能负担反而更低
  • 体验价值：动画提示比静态文字更直观地引导用户"旋转手机"这一操作动作

设计总结

气质定位 · 主色选择 · 关键组件决策

一段话总结

本设计稿以克制、沉浸、精准为气质定调，延续第 1 轮的深蓝水纹 Balatro 官方风格（主色 #4a6bff 蓝 + #ffc857 金），采用等比 scale 缩放（0.51）而非响应式重排作为核心布局策略。 主色选择理由：蓝色贯穿整个 Balatro 品牌识别（Joker 区边框、sidebar 渐变、chips 水彩块），金色是游戏内核心反馈色（金币、目标分、通关奖励），两者在深色背景上对比度强、辨识度高，移动端小尺寸下仍清晰可见。

  • 关键决策 1：scale 缩放容器——在 PC 版 HTML 结构不变的前提下，用 transform: scale(0.51) 整体压缩至移动端横屏视口，零改动地保留所有 GSAP 动画和元素位置关系，是本次适配最核心的架构选择
  • 关键决策 2：安全区在 scale 父级处理——安全区 padding 必须加在 scale 容器的父级（#app），而非 scale 容器内部，否则安全区值也会被 scale 压缩，导致刘海遮挡内容的根本性问题
  • 关键决策 3：按钮触控热区 ::before 扩展——scale 缩放导致按钮实际触控面积随之缩小，通过 CSS ::before 伪元素将触控热区还原至 44pt/dp 最小规格（原始 CSS 约 86px ÷ scale 0.51 ≈ 44pt），在不改变视觉的前提下满足移动端触控规范
